# Radiant Online Browser Compatibility Issues

This file records browser-parity issues observed while running
`./test/test_radiant_online_view_gtest.exe` against the online URL corpus.
The online gtest is expected to keep accumulating pages; pages that expose
crashes, leaks, timeouts, parser errors, script errors, layout failures,
resource failures, rendering failures, or shutdown failures should normally stay
in the corpus and drive local repros plus engine fixes.

Latest corpus status: the 2026-07-02 full online run completed successfully
with 123 gtest batches over 321 URLs. The corpus has since expanded to 461
URLs; the latest 70-page extension passed after fixing the D3/Observable JS
lowering crash, and the Spring/OneTrust batch was rerun after its closure-env
fix. Rows here are compatibility signals, not necessarily failing gtests: many
pages currently complete only because Radiant falls back or recovers.

## Logging Gaps

| Area | Current state | Needed next |
| --- | --- | --- |
| Event callback exceptions | `lambda/js/js_dom_events.cpp` now logs the thrown message for `on*` handlers and `addEventListener` callbacks, so rows can name failures like missing `Element`, missing `docsearch`, or destructuring `undefined`. | Keep this as the minimum standard for runtime compatibility logs. |
| Generic JS call failures | Some rows still only show `js_call_function: not a function, last_fn=...`, which does not identify the property path or receiver that became non-callable. | Enhance `js_call_function` or property-call lowering to log callee/property names and receiver type. This is needed for the IANA, kernel docs, pygame, qemu, and similar rows. |
| Resource fallback | Resource logs now include URL, HTTP status, MIME/content-type in many paths, and unsupported image format names for WebP. Font fallback logs still sometimes show generated `local(...)` URLs as HTTP 404s. | Normalize font-source logging so `local(...)` failures are reported as local font misses, not remote fetch errors, and include the attempted font family/source format. |

## Issue Categories

| Category | Meaning |
| --- | --- |
| HTML parser recovery | Radiant logged an `html5:` parse/recovery diagnostic for markup that a browser accepts and recovers from. These should not crash, but they mark gaps in HTML5 tree-builder compatibility. |
| JS/DOM runtime gap | Page JS hit an unsupported JS feature, DOM API, callable shape, global constructor, or missing third-party bootstrap dependency. The page continued, but browser behavior is incomplete. |
| CSS/resource/image/font fallback | A linked stylesheet, CSS construct, image format, image URL, or font source needed fallback behavior. Fallback is good; detailed tracking tells us what browser features are still missing. |
| HTTP fetch/subresource response | A top-level or linked resource saw 403/404/connection errors. External server behavior may be legitimate, but Radiant must degrade cleanly and shut down without leaks. |
