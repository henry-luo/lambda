# Chromium Editing — Divergence Report

Conformance gauge (informational, never a gate). For every Chromium `assert_selection`
test we can express (input convertible to our schema, ops supported), we replay it through
our editor and compare the **plain text** of our result to Chromium's **expected**, both
stripped identically (tags/markers removed, block ends → newlines). Generated by
`tools/divergence-report.ts`.

## Overall

- **Compared:** 248 tests
- **Exact text match:** 173 (69.8%)
- **Whitespace-only divergence:** 4 (1.6%) — same text, different spacing
- **Text-conformant (exact + whitespace):** 177 (71.4%)
- **Content divergence:** 71 (28.6%) — our result differs in non-whitespace content
- **Errors:** 0 (0.0%) — our editor threw on the sequence

## Key findings — the divergence classes

Reading the content divergences, they cluster into a few stable behaviours where our
PM/Slate-class model intentionally or not-yet matches a browser:

1. **Cross-block backspace-merge (`joinBackward`) — IMPLEMENTED.** `deleteContentBackward`
   at the *start* of a block now merges it into the previous block (descending into a
   previous list’s last item / heading, keeping the target tag). This included fixing a
   tree-corruption bug where a selection spanning only a block boundary was treated as a
   range over the doc root. Lifted `deleting` conformance ~41% → ~61%.
2. **Select-all + delete — IMPLEMENTED.** A multi-block (e.g. whole-document) text range now
   deletes, keeping the start block’s prefix + end block’s suffix and leaving a single
   (often empty) block.
3. **Whitespace / `&nbsp;` fixup — intentional difference (now the dominant divergence).**
   Chrome rewrites runs of spaces and leading/trailing spaces into `\u00A0` to defeat HTML
   collapsing; we store raw text. After (1)–(2), nearly all remaining `inserting` and
   `deleting` divergences are this. A PM-class editor deliberately does not replicate it —
   informational, not a bug.
4. **`insertHTML` of `<div>` blocks — structural.** Chrome flattens pasted `<div>`s inline
   in some contexts; we treat them as blocks (extra newline). Tied to our schema not
   modelling `<div>`.
5. **Caret-movement landing points / forward join.** A few sequences differ because our
   headless caret movement lands on different stops, or because `deleteContentForward` at a
   block *end* does not yet `joinForward`. The next actionable item.

Status: the two structural gaps the previous report flagged — **(1) joinBackward** and
**(2) select-all delete** — are now implemented. Remaining divergences are the intentional
whitespace model (3), the `<div>` schema gap (4), and `joinForward` (5).
