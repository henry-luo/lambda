## 1. Executive Summary

LambdaJS now passes **1,730 of 3,554** official Node.js parallel tests (**48.7%**), locked by the 2026-06-27 `./test/test_node_gtest.exe --update-baseline` run with **0 regressions**. That is **+268 net** over the original Node4 1,462 release-baseline lock and **+316 net** over Node3's 1,414 snapshot — but the headline number hides the real story:

- The previous Node3-era baseline (`test/node/official_baseline.txt`, preserved as `*.stale-1414`) was **stale**: the 2026-06-16 re-measurement showed **77 new passes and 28 regressions** against it, all from unrelated language-conformance work landing without a Node-regression gate. Node4 has now completed two baseline locks: **1,462** on 2026-06-16, then **1,730** on 2026-06-27 after the subsequent stream/crypto/dns/child-process/fidelity work.
- Node3's plan was executed **out of order**. The work that landed (crypto ciphers, Buffer, assert, a real TLS layer, module sub-path resolution, process/os) tracked Phases 6/8/9/10/13. **Phase 7 — the stream state-machine rebuild, identified in Node3 as the highest-reward item — was not rebuilt.** Narrow 2026-06-22 pipe/PassThrough/read-stream/eventNames/addAbortSignal/state-surface/disabled-Duplex/constructor-options/inheritance/backpressure/typed-array/readable-demand/pipeline/readable-push/Transform-callback/readable-ended-capture, 2026-06-23 stream-promises/finished/pipeline-listener/finished-AsyncLocalStorage, and 2026-06-27 Web/BYOB/pipeline-promises/compose slices have landed, but `js_stream.cpp` is still map-based scaffolding rather than a shared stream state machine.
- Because streams were skipped, `http`, `net`, `zlib`, and the crypto cipher all got built **standalone on libuv** instead of on a shared stream core. They still do not compose as real Node streams (basic `pipe()` and `fs.createReadStream()` fixture support now exist, but only a narrow buffered/done `for await` shim, narrow high-water-mark accounting, and no shared I/O stream core). This is now the dominant structural blocker.
- There is still a **stability problem**, but the picture improved: the original 2026-06-16 locked full-suite run had 26 hard crashers + 2 timeouts; the 2026-06-27 baseline update reports **18 crashers, 0 timeouts, and 0 regressions**. Track 0 has cleared the focused `net` crash cluster (`net` is 94/148 with 0 crashers and 0 regressions), converted the focused `child_process` spawn-error crashers into clean behavior (`child_process` is 34/97 with 0 crashers and 0 regressions), cleared the focused `tls` crash cluster (`tls` is 130/214 with 0 crashers and 0 regressions), and fixed the latest cluster/HTTP shutdown regressions before updating the baseline.
- Several modules with **near-complete code score very low** (`assert` 2/14, `buffer` 27/67 in the current focused gate, `util` 7/27). Their gap is *message/semantics fidelity* (exact `ERR_*` codes and Node-format messages), not missing APIs.

**Node4 target: ~1,800 (50%+)** via six tracks, with a stretch to 1,900+. The central thesis remains: **build the real stream core, re-platform the I/O subsystems onto it, then close the wiring / stability / fidelity gaps around it.**

## 4. Root-Cause Re-Analysis

Six structural causes account for the bulk of the locked 2,058 failures plus the current crash pressure.

### RC1 — No real stream state machine (the linchpin)

`js_stream.cpp` models streams as JS objects with marker properties and a mostly synchronous buffer:
- `push()`/`write()` still mutate `__buffer__` and state-map fields directly; narrow write-side `highWaterMark`/`needDrain` callback handling and readable push-side return-value accounting exist, but there is **no real Readable/Writable queue state machine** and no real backpressure propagation.
- the user `_write(chunk, enc, cb)` and `_transform(chunk, enc, cb)` callbacks now have a working narrow callback path, but `_final`/`_flush` and multi-write ordering remain best-effort.
- events still mostly fire **synchronously** inside `push`/`emit` rather than through a complete microtask/nextTick model, so ordering (`data` before `readable`, `end` after drain) is still fragile.
- **only a narrow `Symbol.asyncIterator` shim** — disabled/ended or already-buffered readable sides can complete, but pending `next()` promises, deferred pushes, error rejection, and cleanup are still missing.
- the static surface is `Readable/Writable/Duplex/Transform/PassThrough/pipeline/finished` + `Readable.from` + `addAbortSignal` + a narrow `stream/promises` namespace plus an initial `Readable.prototype.compose` slice. Missing: full `compose`, multi-stage `pipeline`, and `finished` semantics.

51 of 212 stream tests still fail directly, with 0 timeouts and 0 regressions, after the narrow 2026-06-22 pipe/read-stream/eventNames/addAbortSignal/state-surface/disabled-Duplex/constructor-options/inheritance/backpressure/captureRejections/writable-encoding/writable-validation/default-HWM/typed-array/readable-demand/pipeline/EOF-readable/readable-push-backpressure/Transform-callback/setImmediate-args/readable-ended-capture slices, the 2026-06-23 stream-promises/finished/pipeline-listener/finished-AsyncLocalStorage slices, and the 2026-06-27 Web/BYOB/pipeline-promises/compose slices. `test-stream-destroy.js` is now known to be environment-sensitive because it is stream-classified while opening HTTP sockets; it passes with socket access and should not be treated as a destroy lifecycle bug. The remaining failures still block everything in RC2.

### RC2 — I/O subsystems are built standalone on libuv, not on streams

Because there was no stream core to build on, each I/O module reimplemented its own ad-hoc data flow directly over libuv:
- **http** (`js_http.cpp`, 1,329 LOC): `IncomingMessage.body` is a **string**, not a Readable; `ServerResponse` accumulates body in memory; no chunked transfer-encoding; `Agent` pooling is fake.
- **net** (`js_net.cpp`, 1,094 LOC): `Socket` is **not a Duplex**; `setKeepAlive/setNoDelay/ref/unref` are no-ops. `connect()` now validates options and uses libuv DNS/TCP with safer lifetime handling, but the stream contract is still ad-hoc.
- **zlib** (`js_zlib.cpp`): sync only; the streaming `createGzip()` family doesn't exist.
- **crypto cipher**: `Cipher/Decipher` are not Transform streams (only `update`/`final`).

Consequence: nothing composes (`req.pipe(gunzip).pipe(res)` is impossible), and each module independently re-derives buffering/flow bugs. **One real stream core lets all four re-platform.**

### RC3 — Wiring gaps: real implementations not connected

Two high-value capabilities exist but aren't wired:
- **https has no TLS.** `js_https.cpp:39–43` delegates to `js_http_createServer()` on port 443; the comment itself says *"In a full implementation, this would pipe TLS sockets into HTTP parsing."* Meanwhile `js_tls.cpp` is a **real** mbedTLS handshake. Connecting them is wiring, not new crypto.
- **`util.promisify` is partially real.** Node4 C.2 slices replaced the literal `return fn_item` stub with a callback-last Promise wrapper, `util.promisify.custom`, `internal/util.customPromisifyArgs`, Node default first-value resolution, named multi-result objects, and `DEP0174` warning emission. Local module coverage now passes, but the official `test-util-promisify.js` still fails on broader promise/mustCall drain behavior (direct run now reaches one anonymous `mustCall` instead of zero). `callbackify` now has a Promise-to-callback wrapper and official `test-util-callbackify.js` passes, so the remaining utility gap is promisify harness/promise fidelity plus `util.inspect`.

### RC4 — Stability: crashers on error, abort, and invalid-option paths

The original locked full-suite crashers and the 2026-06-18 focused `net`/`tls` reruns were overwhelmingly on **error / abort / invalid-argument / option-validation paths**. The current implementation has JSON-framed IPC, child-side `process.send(message, cb)`, callback invocation after libuv write completion, real `process.once()`, `process.removeListener()` cleanup for once wrappers, and enough default-fork stdio isolation to keep cluster worker failures from leaking into parent official-test output. This recovered `test-child-process-fork-net-server.js`, `test-child-process-send-after-close.js`, `test-child-process-send-cb.js`, `test-cluster-bind-twice.js`, `test-cluster-primary-error.js`, and `test-cluster-uncaught-exception.js` in the 1,730 baseline. Remaining F.1 work: parent-side `child.send()` callback/failure fidelity, robust IPC handle ref/unref behavior, stdio option validation and inheritance parity, signal/disconnect edge cases, and serialization modes beyond JSON framing.

**2026-06-27 regression/baseline round:** Fixed the outstanding Node4 regressions before updating the baseline. ## 9. Appendix: Measurement Methodology

So the numbers above are reproducible:

1. **Build:** `make release` (produces a fresh `lambda.exe`; note it *deletes* the test exe).
2. **Rebuild the runner:** `cd build/premake && make -f test_node_gtest.make config=debug_native`.
3. **Full run:** `./test/test_node_gtest.exe` → prints the Total/Passed/Failed/Crashed/Regressions/Improvements box; writes crashers to `temp/_node_official_crashers.txt`.
4. **Per-module:** `./test/test_node_gtest.exe --modules=<prefix> --timeout=15000`.
5. **Re-baseline:** `make node-update-baseline` (or `./test/test_node_gtest.exe --update-baseline`), which only writes if there are no regressions vs the current baseline.

Tests come from `ref/node/test/parallel/` (3,938 files); the runner filters by the enabled-module table in `test/test_node_gtest.cpp:110+` (dgram/http2/wasi disabled) and the external `test/node/official_skip_list.txt` (71 test entries). PASS = exit 0 and no `Uncaught` in output.
